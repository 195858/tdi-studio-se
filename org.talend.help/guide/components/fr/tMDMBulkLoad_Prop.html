tMDMBulkLoad

Propriétés du tMDMBulkLoad

Famille de composant
Talend MDM
Fonction
Le composant tMDMBulkLoad écrit en masse des données maître structurées XML dans le hub MDM.
Objectif
Ce composant utilise le mode bulk pour écrire des données, de façon à ce que de gros lots de données ou des données très complexes puissent être chargé(e)s rapidement sur le serveur MDM.
Basic settings
Schema et Edit Schema
Un schéma est une description de lignes, i.e., il définit le nombre de champs qui sont traités et passés au composant suivant. Le schéma est soit local (built-in) soit distant dans le Repository.
Cliquez sur Edit Schema pour modifier le schéma.

Si vous modifiez le schéma, il redevient automatiquement built-in.


Cliquez sur le bouton Sync columns afin de récupérer le schéma du composant précédent.
 
 
Built-in : Le schéma sera créé et conservé pour ce composant seulement. Voir également : Paramétrer un schéma built-in du Guide Utilisateur de Talend Integration Suite Studio.
 
 
Repository : Le schéma existe déjà et est stocké dans le Repository. Ainsi, il peut être réutilisé dans divers projets et Jobs. Voir également : Paramétrer un schéma du Repository du Guide Utilisateur de Talend Integration Suite Studio.

Utilisation
Ce composant nécessite toujours un flux d'entrée afin de stucturer des données en XML. Si vos données ne sont pas encore structurées XML, vous devez utiliser des composants tels que le tWriteXMLField afin de transformer ces données en XML. Pour plus d'informations concernant le tWriteXMLField, consultez tWriteXMLField.

Améliorer les chargements de données MDM de masse

Les informations ci-dessous concernent uniquement MDM utilisé avec eXist.

L'analyse XML est un processus qui consomme beaucoup de ressources du processeur et de mémoire et qui n'est pas vraiment compatible avec des ensembles de données particulièrement volumineux. Le Scénario : Charger des enregistrements dans une entité métier donné en exemple pour le composant tMDMBulkLoad possède des limitations parce qu'il ne fonctionne pas avec un ensemble de données très volumineux, pour le moment au moins.

Un scénario alternatif dans lequel vous pouvez traiter l'ensemble de données par itérations de chargement de masse peut être créé comme suit :

Dans un tel scénario, le composant tMDMBulkLoad attend des données XML en entrée. Vous devez formater manuellement ces données entrantes afin de les faire correspondre au schéma de l'entité défini dans MDM Studio. La plupart du temps, les données que vous souhaitez importer sont au format plat et vous devez les transformer en XML.

Puisque l'analyse XML consomme beaucoup de mémoire, vous pouvez régler ce problème en séparant votre fichier source en plusieurs fichiers utilisant le composant tAdvancedFileOutputXML. Pour ce faire, sélectionnez l'option Split output in several files dans la vue Advaced settings du composant puis configurez les lignes dans chaque fichier de sortie via une variable de contexte (context.chunkSize), par exemple.

Le schéma XML que vous devez définir dans l'éditeur XML du composant doit être une correspondance exactea de l'entité métier définie dans le Studio MDM. Le schéma XML dans l'éditeur doit représenter un seul élément racine <root> qui contient tous les autres éléments, Afin que vous puissiez effectuer une boucle sur chaque élément. Le chemin d'accès au fichier doit être défini dans un dossier temporaire.

Utilisez un composant tFileList pour lire tous les fichiers XML ayant été créés. Ce composant vous permet de paralléliser le processus. Connectez-le à un composant tFileInputXML à l'aide d'un lien Iterate.

Pour le lien Iterate, il est recommandé de configurer autant de processus d'exécution qu'il y a de coeurs à votre ordinateur. Vous pouvez faire ceci en utilisant Runtime.getRuntime().availableProcessors()

Le composant tFileInputXML va lire les données des fichiers XML que vous avez créés, en définissant une boucle sur les éléments et en récupérant tous les noeuds déjà formatés en XML. Vous devez ensuite cocher la case Get Nodes.

Soyez sûr d'avoir configuré la taille de commit avec la même valeur que celle définie dans le composant tAdvancedfileOutputXML, la variable de contexte context.chunkSize.

Le composant tFiledelete dans ce scénario va supprimer toutes les données temporaires à la fin du Job.
