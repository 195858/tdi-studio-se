tStandardizeRow

This component will be available in the Palette of the studio on the condition that you have subscribed to the relevant edition of Talend Data Quality.

tStandardizeRow properties

Component family
Data quality
Function
tStandardizeRow tokenizes the data flow it has received from the preceding component and applies user-defined rules to parse the data. Based on this parsing, this component normalizes and writes these data in a separate XML data flow and tags them using the user-defined rule names. The standardization option adds a supplementary column to the output flow where the normalized data are then standardized.
ANTLR is used to parse and tokenize the incoming data. For further information about ANTLR, see the site
http://www.antlr.org/


This component does not make any changes on your raw data.

Purpose
tStandardizeRow normalizes the incoming data in a separate XML data flow. This alllows you to separate, even standardize if need be, the rule-compliant data from the noncompliant data.

Conversion rules
Complete this table to define the rules you need to apply. The table columns to be filled are:
- Name: in this column, name the rule you need to use. This name is also used as the XML tag name and the token name to label the incoming data identified by this rule.
- Type: in this column, select the type of the rule you need to apply. For further information about the rule types available, see section Rule types.
- Value: in this column, type in the rule you need to define.

When several rules of the same type use the identical rule name in this table, they are executed in top-down order, so arrange them properly in order to obtain the best result.

Rule types

Two groups of rule types are provided: the basic rule types and the advanced rule types.

      • Basic rule types: Enumeration, Format and Combination. Rules of these types are composed with some given ANTRL symbols.
      • Advanced rule types: Regex, Index and Shape. Rules of these types match the tokenized data and standardize them when needed.

The following three tables successively present detailed information about the basic types, the advanced types and the ANTRL symbols used by the basic rule types. These three tables help you to complete the Conversion rules table in the Basic settings of this component.

For basic rule types:

Enumeration
A rule of this type provides a list of optional criteria
RuleName: Gender
RuleValue: " `Male' | `Female'"
- Each option must be put in a pair of single quotation marks.

- Defined options must be separated by the | symbol.

- The rule name must start with upper case
Format
(Rule name starts with upper case)
A rule of this type defines the composition of a string by specifying the type of each element of this string.
RuleName: ProductNumber
RuleValue: " DIGIT DIGIT LETTER `-` DIGIT DIGIT "

This rule matches strings like, 13A-74
- Only the pre-defined element types (ANTLR tokens) are accepted to define the string elements. They are: DIGIT, LETTER, UPPERCASEDLETTER, CAPITALIZEDWORD, WORD, INTEGER, DECIMALFLOAT, FRACTIONFLOAT.
Format
(Rule name starts with lower case)
A rule of this type defines the composition of a string by specifying the type of each element of this string.
In addition to the pre-defined element types, an Enumeration rule is applicable to define the type of a string element.
RuleName: length
RuleValue: "DECIMALFLOAT LengthUnit "

The rule matches strings like 1.4 cm, where the DecimalFloat is one of the pre-defined element type and LengthUnit is an Enumeration rule defining " `inch' | 'cm' ".

For advanced rule types:

Advanced Rule Type
Usage
Example
Warning
Regex
A rule of this type uses regular expressions to match the incoming data tokenized by ANTLR.
RuleName: ZipCode
RuleValue: "\\d\\d\\d\\d\\d"

The rule matches strings like "92150"
Regular expressions must be Java compliant.
Index
A rule of this type uses a synonym index as reference to search for the matched incoming data.
For further information about this synonym index, see section tSynonymOutput.
A scenario is available in section Scenario 2: Standardizing the addresses and city names saved in the client data on page 1043.
In Windows, the backslashes \ need to be doubled or replaced by slashes / if the path is copied from the file system.
Shape
A rule of this type uses pre-defined tokens along with the established Regex or Index rules or both to match the incoming data.
RuleName: Address
RuleValue: "<INTR><WORD><StreetType>"

This rule matches the addresses like 12 main street, where StreetType is an Index rule which is used for the current data flow.
- Pre-defined tokens are: INT for integer; WORD for word; WORD+ for literals of several words; CAPWORD for capitalized word; FlOAT for Decimal/Fraction Float.

- Only the contents put in < > are recognizable. In the other cases, the contents are considered as error or are omitted.
