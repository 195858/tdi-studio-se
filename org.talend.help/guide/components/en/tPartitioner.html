tPartitioner

Warning

This component will be available in the Palette of the Integration perspective of your studio on the condition that you have subscribed to one of the Talend Platform solutions or Big Data solutions.

The parallelization components work closely with each other to accomplish parallel execution on given processes. The tPartitioner component dispatches the input records into a specific number of threads; the tCollector component sends these threads to its following components for parallel execution; the tDepartitioner component regroups the outputs of the processed parallel threads; the tRecollector component captures the output of a given tDepartitioner component and sends the captured data to the next component.

tPartitioner Properties

Component family

Orchestration

 

Function

This component splits the input records into a given number of threads which tCollector sends for parallel execution.

Purpose

This component partitions the input data before tCollector can transfer them to the parallel execution processes.

Basic settings

Schema and Edit Schema

A schema is a row description, i.e., it defines the number of fields to be processed and passed on to the next component. The schema is either Built-in or stored remotely in the Repository.

Click Edit Schema to make changes to the schema. Note that if you make changes, the schema automatically becomes built-in.

Click Sync columns to retrieve the schema from the previous component connected in the Job.

 

 Number of Child Threads

Enter the number of threads you want to split the input records up into.

We recommend that this number be N-1 where N is the total number of CPUs or cores on the machine processing the data.

 

Buffer Size

Enter the number of rows to be processed before the memory is freed.

This is the number of rows to cache for each of the threads generated.

 

Use a key hash for partitions

Select this check box to use the hash mode to dispatch the input records into threads.

Once selecting it, the Key Columns table appears, in which you set the column(s) you want to apply the hash mode on. In the hash mode, the records meeting the same criteria are dispatched into the same threads.

If you leave this check box clear, the dispatch mode is Round-robin, meaning records are dispatched one-by-one to each thread, in a circular fashion, until the last record is dispatched. Be aware that this mode cannot guarantee that records meeting the same criteria go into the same threads.

Advanced settings

tStatCatcher Statistics

Select this check box to collect the log data at the component level.

Usage

This component should be put after the input component(s) and before tCollector.

This component uses and can only use the Trigger > Start link to connect to tCollector.

Global Variables

 

NB_LINE: indicates the number of lines processed. This is available as an After variable

For further information regarding global variables, see Talend Data Integration Studio User Guide.

Connections

Scenario: Sorting out the duplicate rows in the parallel fashion

The Job in this scenario uses essentially the parallelization components to sort out the duplicates from the customer records.

This scenario is simplified in order to focus the explanations on the collaboration and the functioning mechanism between each other of the four parallelization components.

Linking the components

  1. In the Integration perspective of your studio, create an empty Job, named Parallelization for example, from the Job Designs node in the Repository tree view.

    For further information about how to create a Job, see the user guide of your integration studio.

  2. Drop the following components onto the workspace: tRowGenerator, tPartitioner, tCollector, tUniqRow, two tDepartitioner components, two tRecollector components, two tSortRow components and two tLogRow components.

    The tRowGenerator component generates the data to be transformed and processed in parallel. In the real-world case, you can use other components such as tMysqlInput or tMap in the place of tRowGenerator to design a sophisticated process to prepare your data to be processed.

  3. Label the two tDepartitioner components as Unique and Duplicates, respectively.

    For further information about how to label a component, see the user guide of your integration studio.

  4. Do the same for the two tLogRow components.

  5. Connect tPartitioner to tCollector and to each of the two tRecollector components, using the Trigger > Starts link.

  6. Connect the other components using the Row > Main link. Among them, use Row > Uniques to connect tUniqRow to the tDepartitioner component labelled Unique and use Row > Duplicates to connect tUniqRow to the tDepartitioner component labelled Duplicates.

Splitting the input records

Procedure 18.1. Configuring the input records

  1. Double-click tRowGenerator to open its editor.

  2. Click the button five times to add five rows to the Schema table and rename them as id, firstname, lastname, city and age, respectively.

  3. In the Type column, select the Integer type for the id row and the age row.

  4. In the Functions column, select Numeric.sequence for the id row and the age row.

  5. Select TalendDataGenerator.getFirstName for the firstname row, TalendDataGenerator.getLastName for the lastname row and TalendDataGenerator.getUsCity for the city row.

  6. In the Number of Rows for RowGenerator field, enter the number of the input rows you want to generate. In this example, enter 10000. In a real-world case, this number is ofter much larger so that the parallelization components are ideal to handle the volume.

  7. Click OK to validate these changes and accept the propagation prompted by the pop-up dialog box.

Procedure 18.2. Configuring the tPartitioner component

  1. Double-click tPartitioner to open its Component view.

  2. In the Number of Child Threads field, enter the number of the threads you want to split the input records up into. In this example, enter 4.

  3. If required, change the value in the Buffer Size field to adapt the memory capacity. In this example, we leave the default one.

  4. Select the Use a key hash for partitions check box to display the Key Columns table.

  5. Click the button three times to add three rows and sequentially, select firstname, lastname and city for the three rows. They will be the key columns where the hash mode is applied to dispatch, to the same threads, the records meeting the same hash criteria.

  6. Click the button next to Edit schema to open the schema editor.

  7. Select all the rows in the tPartitioner component schema table using Ctrl or Shift.

    Then the on the toolbar is activated.

  8. Click to copy the schema selected.

Deduplicating the input records

Procedure 18.3. Configuring tCollector

  1. Double-click tCollector to open its Component view.

  2. Click the button next to Edit schema to open the schema editor.

  3. Press Ctrl+V to paste the tPartitioner component schema.

  4. Click OK to validate these changes and accept the propagation prompted by the pop-up dialog box.

Procedure 18.4. Configuring tUniqRow

  1. Double-click tUniqRow to open its Component view.

  2. In the Unique key table, the schema appears automatically. Select the check boxes corresponding to firstname, lastname and city in the Key attribute column. The Job is based on the three columns to sort out the duplicates.

    If the schema does not appear, click the Sync columns button to retrieve the schema from the preceding component.

Procedure 18.5. Verifying tDepartitioner

  1. Double-click the tDepartitioner component labelled Unique to open its Component view. This component is connected through the Unique link.

  2. If required, change the values in the Buffer Size field to adapt the memory capacity. In this example, we leave the default value.

  3. Click the button next to Edit schema to open the schema editor, then, check all of the columns you need to output appear in the schema table of tDepartitioner. In this scenario, we output all the columns received from its preceding components.

  4. Do the same to check the tDepartitioner component labelled Duplicates.

Outputting the unique rows

Procedure 18.6. Capturing the unique rows

  1. Double-click the tRecollector component, linked to the tLogRow component labelled Unique, to open the Component view of this tRecollector.

  2. Click the button next to Edit schema to open the schema editor, then, paste the tPartitioner schema we copied earlier when we were configuring tPartitioner.

    This schema should be consistent with that of the tDepartitioner component offering data to the current tRecollector. In this example, the tDepartitioner component to be used has the same schema as the tPartitioner component. If this is not your case in the real practice, you have to copy-paste the schema of the tDepartitioner component you want to use.

  3. Click OK to validate these changes and accept the propagation prompted by the pop-up dialog box.

  4. In the Linked Departitioner field, select the tDepartitioner component which offers the unique data. In this example, it is the tDepartitioner component labelled Unique. Therefore, select tDepartitioner_1 - Unique.

Procedure 18.7. Sorting the unique rows

  1. Double-click the tSortRow component, linked to the Unique tLogRow component, to open the Component view of this tSortRow.

  2. Under the Criteria table, click the button three times to add three rows to the table.

  3. In the Schema column column, select, for each row, the schema column to be used as the sorting criterion. In this example, select firstname, lastname and city, sequentially.

  4. In the Sort num or alpha? column, select alpha for all the three rows, because the data of the firstname, the lastname and the city schema columns are all letters.

  5. In the Order asc or desc column, select asc for all the three rows.

  6. Double-click the tLogRow component labelled Unique to open its Component view.

  7. In the Mode area, select Table (print values in celles of a table).

Outputting the duplicate rows

You need to configure the tRecollector, the tSortRow and the tLogRow (labelled Duplicate) components you linked together to output the duplicate rows.

  • Repeat, on these components, the operations you performed earlier to output the unique rows, but select tDepartitioner_2 - Duplicate instead, in the Linked tDepartitioner field of the Basic settings tab view of the tRecollecter component to be used for the duplicate rows.

Executing the Job

Then you can press F6 to run this Job.

Once done, the Run view is opened automatically, where you can check the execution result.
